# UML Class Diagram Relationships: একদম সহজ ভাষায় গাইড

একটি সফটওয়্যারে কখনো একটি মাত্র ক্লাস দিয়ে সব কাজ করা যায় না। বিভিন্ন ক্লাসকে একসাথে কাজ করতে হয়। UML ক্লাস ডায়াগ্রামে **Relationships** বা সম্পর্কগুলোর কাজ হলো— একটি ক্লাস অন্য ক্লাসের সাথে কীভাবে এবং কতটুকু গভীরভাবে যুক্ত, তা লাইনের মাধ্যমে ফুটিয়ে তোলা।

মনে করুন আপনি প্রোগ্রামিং বা সফটওয়্যার ডিজাইনে একদম নতুন। নিচে ৬টি প্রধান সম্পর্ককে বাস্তব উদাহরণ এবং কোড সহ সহজ বাংলায় ব্যাখ্যা করা হলো:

---

## ১. Inheritance / Generalization (উত্তরাধিকার সম্পর্ক - "Is-A")

### সহজ কথায় মানে:
যখন একটি ক্লাসের সমস্ত বৈশিষ্ট্য (ভেরিয়েবল ও ফাংশন) অন্য একটি ক্লাসের মধ্যে স্বয়ংক্রিয়ভাবে চলে আসে। এটিকে **"Is-A" (হয় একটি)** সম্পর্ক বলা হয়।

### বাস্তব উদাহরণ:
* **গাড়ি** (`Car`) হলো এক ধরনের **যানবাহন** (`Vehicle`)।
* **বিড়াল** (`Cat`) হলো এক ধরনের **প্রাণী** (`Animal`)।
* এখানে `Vehicle` হলো পিতা (Parent) এবং `Car` হলো সন্তান (Child)।

### ড্রয়িং:
```text
  [ Car ] ───────▷ [ Vehicle ]
```
*(লাইনটি সলিড বা সোজা এবং তীরটি ফাঁপা ত্রিভুজ, যা প্যারেন্ট ক্লাসের দিকে নির্দেশ করে).*

### কোডে কেমন দেখায়:
```java
// পিতা (Superclass)
class Vehicle {
    String brand;
    void start() {
        System.out.println("Engine started");
    }
}

// সন্তান (Subclass) - extends ব্যবহারের মাধ্যমে Inheritance হয়
class Car extends Vehicle {
    int doors = 4;
}
```

## ২. Realization / Implementation (চুক্তি বা বাস্তবায়ন সম্পর্ক)

### সহজ কথায় মানে:
যখন একটি ক্লাসের ভেতর কী কী কাজ করা যাবে তার একটি তালিকা বা নিয়ম (Interface) দেওয়া থাকে এবং অন্য কোনো ক্লাস সেই নিয়মগুলো বাস্তবে কীভাবে কাজ করবে তা কোড লিখে বুঝিয়ে দেয়।

### বাস্তব উদাহরণ:
* **রিমোট কন্ট্রোল** (`RemoteControl`) একটি ইন্টারফেস বা নিয়ম। এতে বোতাম আছে যেমন `powerOn()`।
* **সনি টিভি** (`SonyTV`) বা **স্যামসাং টিভি** (`SamsungTV`) ক্লাসগুলো বাস্তবে এই বোতাম চাপলে কী হবে তা কোড লিখে বাস্তবায়ন করে।

### ড্রয়িং:
```text
  [ SonyTV ] ╌╌╌╌╌╌╌▷ [ RemoteControl ]
```
*(লাইনটি ড্যাশড বা ডটেড এবং তীরটি ফাঁপা ত্রিভুজ).*

### কোডে কেমন দেখায়:
```java
// নিয়ম বা চুক্তি (Interface)
interface RemoteControl {
    void powerOn();
}

// নিয়ম বাস্তবায়নকারী ক্লাস (Concrete Class) - implements ব্যবহার করা হয়
class SonyTV implements RemoteControl {
    public void powerOn() {
        System.out.println("Sony TV is now ON");
    }
}
```

## ৩. Association (সাধারণ যোগাযোগ সম্পর্ক - "Has-A")

### সহজ কথায় মানে:
যখন দুটি ক্লাস একে অপরকে চেনে বা সরাসরি যোগাযোগ করে। এটি মূলত দুই প্রকার:
1. **একমুখী (Directed):** শুধু ক্লাস A ক্লাস B-কে চেনে বা ব্যবহার করে, কিন্তু B জানে না A কে।
2. **উভমুখী (Bidirectional):** উভয় ক্লাসই একে অপরকে চেনে।

### বাস্তব উদাহরণ:
* **ক্রেতা** (`Customer`) এবং **অর্ডার** (`Order`)। ক্রেতার কাছে তার অর্ডারের তথ্য থাকে। এটি একটি "Has-A" (কাছে আছে) সম্পর্ক।

### ড্রয়িং (একমুখী):
```text
  [ Customer ] ───────> [ Order ]
```
*(সরাসরি সোজা লাইন ও সাধারণ তীর, যা ব্যবহৃত ক্লাসের দিকে নির্দেশ করে).*

### কোডে কেমন দেখায়:
```java
class Order {
    String orderId;
}

class Customer {
    String name;
    Order myOrder; // Customer ক্লাসের ভেতরে Order ক্লাসের অবজেক্ট ভেরিয়েবল হিসেবে রাখা হয়েছে
}
```

## ৪. Aggregation (শিথিল বা দুর্বল মালিকানা - "Weak Has-A")

### সহজ কথায় মানে:
এটি এক ধরনের অ্যাসোসিয়েশন, তবে এখানে মালিকানার বিষয় থাকে। এখানে মূল ক্লাসের অধীনে কিছু অংশ বা অন্য ক্লাসের অবজেক্ট থাকে। তবে সবচেয়ে গুরুত্বপূর্ণ বিষয় হলো: **যদি মূল ক্লাসটি ধ্বংস হয়ে যায়, তবুও ভেতরের অন্য ক্লাসগুলোর অবজেক্ট স্বাধীনভাবে বেঁচে থাকতে পারে।**

### বাস্তব উদাহরণ:
* একটি **লাইব্রেরি** (`Library`) এবং **বই** (`Book`)।
* যদি লাইব্রেরিটি চিরতরে বন্ধ হয়ে যায় বা ধ্বংস হয়ে যায়, বইগুলো কিন্তু ধ্বংস হয়ে যায় না। বইগুলো অন্য লাইব্রেরিতে নিয়ে যাওয়া সম্ভব।

### ড্রয়িং:
```text
  [ Book ] ───────◇ [ Library ]
```
*(সরাসরি লাইন এবং ফাঁপা ডায়মন্ড চিহ্নটি মূল বা মালিক ক্লাসটির গায়ে লাগানো থাকে).*

### কোডে কেমন দেখায়:
```java
class Book {
    String title;
}

class Library {
    String name;
    List<Book> books; // বইগুলো লাইব্রেরির বাইরে তৈরি হয়ে এখানে পাস করা হয়
    
    Library(List<Book> books) {
        this.books = books;
    }
}
```

## ৫. Composition (দৃঢ় বা শক্তিশালী মালিকানা - "Strong Has-A")

### সহজ কথায় মানে:
এটিও মালিকানার সম্পর্ক, তবে অত্যন্ত শক্তিশালী। এখানে মূল ক্লাস এবং তার ভেতরের অন্য ক্লাসের জীবনকাল একসাথে বাধা। **যদি মূল ক্লাসটি ধ্বংস হয়ে যায়, তবে ভেতরের অন্য ক্লাসের অবজেক্টও সাথে সাথে ধ্বংস হয়ে যাবে। তাদের নিজস্ব আলাদা কোনো অস্তিত্ব নেই।**

### বাস্তব উদাহরণ:
* **মানুষ** (`Human`) এবং **হৃদপিণ্ড** (`Heart`)।
* মানুষটির মৃত্যু হলে হৃদপিণ্ডটিরও মৃত্যু ঘটে। মানুষের শরীর ছাড়া হৃদপিণ্ড স্বাধীনভাবে বেঁচে থাকতে পারে না।

### ড্রয়িং:
```text
  [ Heart ] ───────◆ [ Human ]
```
*(সরাসরি লাইন এবং ভরাট কালো ডায়মন্ড চিহ্নটি মূল বা মালিক ক্লাসটির গায়ে লাগানো থাকে).*

### কোডে কেমন দেখায়:
```java
class Heart {
    void beat() { }
}

class Human {
    Heart heart;
    
    Human() {
        // Human তৈরি হওয়ার সময় ভেতরেই Heart তৈরি হচ্ছে
        this.heart = new Heart(); 
    }
    // Human অবজেক্ট যখন মেমোরি থেকে ডিলিট হবে, heart-ও ডিলিট হয়ে যাবে
}
```

## ৬. Dependency (সাময়িক নির্ভরশীলতা - "Uses-A")

### সহজ কথায় মানে:
সবচেয়ে দুর্বল সম্পর্ক। যখন একটি ক্লাস অন্য একটি ক্লাসকে স্থায়ীভাবে নিজের কাছে জমা রাখে না, বরং সাময়িক কোনো কাজের জন্য ব্যবহার করে। যেমন মেথডের প্যারামিটার হিসেবে বা লোকাল ভেরিয়েবল হিসেবে।

### বাস্তব উদাহরণ:
* **ড্রাইভার** (`Driver`) এবং **গাড়ি** (`Car`)। 
* ড্রাইভার স্থায়ীভাবে গাড়ির মালিক নাও হতে পারেন। তিনি শুধু সাময়িকভাবে গাড়িটি চালানোর জন্য (`drive(Car car)`) কার অবজেক্টটি ব্যবহার করেন। কাজ শেষ হলে তাদের আর সরাসরি যোগাযোগ থাকে না।

### ড্রয়িং:
```text
  [ Driver ] ╌╌╌╌╌╌╌> [ Car ]
```
*(ড্যাশড বা ডটেড লাইন ও সাধারণ তীর, যা ব্যবহৃত ক্লাসের দিকে নির্দেশ করে).*

### কোডে কেমন দেখায়:
```java
class Car {
    void move() { }
}

class Driver {
    // Car কিন্তু ক্লাসের মেম্বার ভেরিয়েবল হিসেবে নেই। 
    // এটি শুধু drive মেথডের ভেতর সাময়িক প্যারামিটার হিসেবে ব্যবহৃত হচ্ছে।
    void drive(Car car) {
        car.move();
    }
}
```

## সম্পর্কের পার্থক্য বোঝার জন্য ৩টি জাদুকরী প্রশ্ন (Golden Rules)

আপনি যখন নিজে কোড ডিজাইন করবেন, তখন নিজেকে এই প্রশ্নগুলো করবেন:

1. **এটি কি অন্যটির একটি রূপ? (Is-A):** 
   - উত্তর 'হ্যাঁ' হলে ──▷ **Inheritance** ব্যবহার করুন। (যেমন: Dog is an Animal)
2. **মালিকানা কি অত্যন্ত দৃঢ়? মূল অবজেক্ট ধ্বংস হলে কি অন্যটি মারা যাবে?:**
   - উত্তর 'হ্যাঁ' হলে ──◆ **Composition** ব্যবহার করুন। (যেমন: House ও Room)
   - উত্তর 'না' কিন্তু সম্পর্ক আছে হলে ──◇ **Aggregation** ব্যবহার করুন। (যেমন: Department ও Teacher)
3. **যোগাযোগটি কি শুধু সাময়িক মেথড কলের জন্য?:**
   - উত্তর 'হ্যাঁ' হলে ──> **Dependency** ব্যবহার করুন।
