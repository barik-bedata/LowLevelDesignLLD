# UML Class Diagram-এর কম্পোনেন্ট ও মাল্টিপ্লিসিটি (Multiplicity)

ইউএমএল ক্লাস ডায়াগ্রাম প্রধানত ৩টি মৌলিক কম্পোনেন্ট বা উপাদান নিয়ে গঠিত:
১. **Class (ক্লাস)**
২. **Multiplicity / Cardinality (সংখ্যাগত সম্পর্ক)**
৩. **Relationships (সম্পর্ক বা সংযোগকারী লাইন)**

নিচে প্রতিটি কম্পোনেন্ট সহজ বাংলায় বিস্তারিত আলোচনা করা হলো।

---

## ১. Class Component (ক্লাসের গঠন)

একটি ক্লাস বক্সের ভেতরে ৩টি মূল অংশ থাকে:

```text
┌────────────────────────────────────────┐
│               ClassName                │ <── ১. ক্লাসের নাম (Class Name)
├────────────────────────────────────────┤
│ - attribute1: type                     │ <── ২. প্রোপার্টি/ডাটা (Attributes)
│ + attribute2: type                     │
├────────────────────────────────────────┤
│ + method1(param): returnType           │ <── ৩. ফাংশন/কাজ (Methods)
│ - method2(): returnType                │
└────────────────────────────────────────┘
```

### অ্যাক্সেস মডিফায়ার (Access Modifiers):
Attributes ও Methods এর অ্যাক্সেস লেভেল বোঝাতে ৪টি চিহ্ন ব্যবহৃত হয়:
* **`-` (Private):** শুধু ওই ক্লাসের ভেতরেই অ্যাক্সেসযোগ্য।
* **`+` (Public):** যেকোনো ক্লাস থেকে অ্যাক্সেস করা যাবে।
* **`#` (Protected):** নিজের ক্লাস এবং তার চাইল্ড ক্লাস থেকে অ্যাক্সেসযোগ্য।
* **`~` (Package/Default):** একই প্যাকেজের ভেতরের ক্লাসগুলো অ্যাক্সেস করতে পারবে।

---

## ২. Multiplicity / Cardinality (সংখ্যাগত সম্পর্ক)

দুটি ক্লাসের মধ্যে ডিরেকশনাল লাইনের দুই প্রান্তে কিছু সংখ্যা বা চিহ্ন লেখা থাকে। একে **Multiplicity** বা **Cardinality** বলা হয়। এর মাধ্যমে বোঝানো হয় একটি ক্লাসের একটি অবজেক্ট অন্য ক্লাসের কতটি অবজেক্টের সাথে যুক্ত থাকতে পারবে।

নিচে জনপ্রিয় মাল্টিপ্লিসিটি চিহ্ন এবং তাদের অর্থ উদাহরণসহ দেওয়া হলো:

### ক. `1` (ঠিক একটি / Exactly One)
* **অর্থ:** সম্পর্কটি বাধ্যতামুলক এবং ঠিক ১টি অবজেক্টের সাথেই হতে হবে। ১টির কম বা বেশি হওয়া যাবে না।
* **উদাহরণ:** একজন নাগরিকের (`Citizen`) ঠিক একটি ন্যাশনাল আইডি কার্ড (`NationalID`) থাকে।
* **ড্রয়িং:**
  ```text
  [ Citizen ] 1 ────────────────── 1 [ NationalID ]
  ```

---

### খ. `0..1` (শূন্য অথবা একটি / Zero or One)
* **অর্থ:** এটি ঐচ্ছিক (Optional)। অবজেক্ট থাকতেও পারে, নাও থাকতে পারে (সর্বোচ্চ ১টি)।
* **উদাহরণ:** একজন চাকরিজীবীর (`Employee`) একটি অফিস কার (`OfficeCar`) থাকতে পারে, আবার নাও থাকতে পারে।
* **ড্রয়িং:**
  ```text
  [ Employee ] 1 ────────────────── 0..1 [ OfficeCar ]
  ```

---

### গ. `*` অথবা `0..*` (শূন্য অথবা অনেক / Zero or Many)
* **অর্থ:** এটিও ঐচ্ছিক। কোনো অবজেক্ট নাও থাকতে পারে, আবার অনির্দিষ্ট সংখ্যক অনেক অবজেক্টও থাকতে পারে।
* **উদাহরণ:** একজন ক্রেতা (`Customer`) কোনো অর্ডার নাও দিতে পারেন, আবার অনেকগুলো অর্ডারও করতে পারেন।
* **ড্রয়িং:**
  ```text
  [ Customer ] 1 ────────────────── * [ Order ]
  ```

---

### ঘ. `1..*` (কমপক্ষে একটি বা তার বেশি / One or Many)
* **অর্থ:** সম্পর্কটি বাধ্যতামুলক। কমপক্ষে ১টি অবজেক্ট থাকতেই হবে, এবং সর্বোচ্চ অনেক হতে পারে।
* **উদাহরণ:** একটি কলেজ বা ডিপার্টমেন্টে (`Department`) কমপক্ষে ১ জন শিক্ষক (`Teacher`) থাকতেই হবে, একাধিকও থাকতে পারেন। শিক্ষক ছাড়া ডিপার্টমেন্টের অস্তিত্ব কল্পনা করা যায় না।
* **ড্রয়িং:**
  ```text
  [ Department ] 1 ────────────────── 1..* [ Teacher ]
  ```

---

### ঙ. `n..m` (নির্দিষ্ট সীমার মধ্যে / Specific Range)
* **অর্থ:** অবজেক্টের সংখ্যা একটি নির্দিষ্ট সীমার ভেতরে থাকতে হবে (যেমন `২ থেকে ৪টি`)।
* **উদাহরণ:** একটি গাড়ির (`Car`) চাকায় `2..4` বা সাধারণত `4` টি হুইল থাকে। অথবা একটি ফুটবল টিম খেলতে গেলে মাঠে কমপক্ষে `11` জন খেলোয়াড় থাকতে হবে।
* **ড্রয়িং:**
  ```text
  [ Car ] 1 ────────────────── 4 [ Wheel ]
  ```

---

## ৩. Relationships Component (সম্পর্কসমূহ)

লাইনগুলোর দিক এবং চিহ্ন মনে রাখার জন্য সংক্ষেপ চিত্র:

```text
১. Inheritance (চাইল্ড থেকে প্যারেন্ট):
   [ ChildClass ] ────────▷ [ ParentClass ]

২. Realization (ক্লাস থেকে ইন্টারফেস):
   [ Class ] ╌╌╌╌╌╌╌▷ [ Interface ]

৩. Association (ব্যবহারকারী থেকে ব্যবহৃত):
   [ User ] ────────> [ TargetClass ]

৪. Aggregation (অংশ থেকে মূল - শিথিল মালিকানা):
   [ Part/Component ] ────────◇ [ Whole/Container ]

৫. Composition (অংশ থেকে মূল - দৃঢ় মালিকানা):
   [ Part/Component ] ────────◆ [ Whole/Container ]

৬. Dependency (সাময়িক নির্ভরশীলতা):
   [ ClientClass ] ╌╌╌╌╌╌╌> [ SupplierClass ]
```

### সম্পর্কের সাথে মাল্টিপ্লিসিটির একটি বাস্তব উদাহরণ:

ধরুন, একটি লাইব্রেরি সিস্টেম:
* ১টি লাইব্রেরিতে (`Library`) কমপক্ষে ১টি বা তার বেশি বই (`Book`) থাকতে পারে (`1..*`)।
* আবার ১টি বই একই সাথে ১টির বেশি লাইব্রেরির অংশ হতে পারে না (বাস্তব লাইব্রেরির ক্ষেত্রে `1`)।

```text
  [ Library ] 1 ────────────────── 1..* [ Book ]
```
*(এর মানে: লাইব্রেরি ধ্বংস হলে সব বই ধ্বংস হয়ে যাবে না, অর্থাৎ এটি একটি Aggregation সম্পর্ক).*
